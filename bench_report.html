What changed

Dispatcher task removed. The previous design routed every fanout publish through a per-slot mpsc plus a dedicated tokio task whose only job was to call broadcast::Sender::send. The intent was to keep the router O(1) in subscriber count by deferring the O(N) waker walk onto a separate task — but broadcast::send is already a synchronous, non-blocking call: it writes into the ring buffer (lagged receivers get a Lagged error on their next recv) and walks the waker list inline. The extra mpsc hop and dispatcher task therefore added overhead without reducing the total work.

Router now calls sender.send(delivery) directly with no .await. The hot path in router.rs has zero yields for fanout. The net effect is visible in the slope: the per-added-subscriber cost collapses from ≈86 ns to ≈6 ns because the extra hop was paying a fixed cost per publish that grew with N (mpsc wake + dispatcher wake + task switch).

Structural cleanup (free with the refactor):

  • Slot::Broadcast variant no longer carries dispatch_tx; only sender and an explicit sub_count remain.
  • Subscribe no longer spawns a dispatcher task. One less task per fanout slot.
  • Slot cleanup on disconnect moved from an O(M) index.retain scan to O(1) via a new slot_keys: U64Map<SlotId, IndexKey> reverse index.
  • The index key is now (Box<str>, Option<Box<str>>) instead of (Box<str>, Box<str>), making the fanout/queue-group distinction explicit.

Small-N trade-off

Between 1 and 10 subscribers, fanout latency regressed from ~247 ns to ~274 ns (1 sub) and ~276 ns to ~499 ns (10 subs). The cause is the loss of pipelining: the old dispatcher pattern let router and dispatcher run on two different tokio worker threads, so while the router prepared publish N+1, the dispatcher was still waking subscribers for publish N. With the dispatcher gone, the router does both sequentially.

For small N the per-wake cost is tiny, so the loss of parallel overlap shows up as a few hundred ns of extra latency. For large N the per-wake cost dominates and the saved hop wins decisively (13.9× at 1000 subs). The curve is now monotonically close to linear in N with a very small slope, which matches the objective of flat fanout latency scaling.

Queue group and wildcard

queue_group_4_members_e2e: +10.4 % (317 ns vs 287 ns). The queue-group path did not change (still kanal MPMC with router .await). The small regression is memory-layout noise from the reshaped SlotMap struct (added slot_keys field, queue_group typed as Option).

wildcard_routing_e2e: +28.8 % (299 ns vs 232 ns). This bench is essentially fanout_e2e with 1 subscriber on a wildcard pattern, so the small-N regression story applies.

Methodology

  • Criterion 0.8, 100 samples per measurement, 3 s warmup + 5 s collection.
  • Each iteration: publish one message, synchronize on a sequence tracker via AtomicU64 + Notify, then yield.
  • All tasks spawned on the same multi-threaded tokio runtime (Criterion's default async_tokio).
  • Payload: 128 bytes. Latency is pure in-process router cost — no gRPC, no TLS, no TCP.
  • Both fanout_e2e (waits for all N subscribers to read) and fanout_publish_only (waits for a single sentinel subscriber, ignores N background drains) were run so publisher-side cost can be separated from consumer-side wake-up cost.
  • Test suite: 18 unit tests (including new fanout_send_never_blocks_router and lagged_subscriber_gets_lagged_error), 6 integration tests, 1 doctest — all pass.
  • cargo clippy --workspace --all-targets -- -D warnings: clean.
