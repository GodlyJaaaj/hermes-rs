Simplification pass — what shipped

On top of removing the per-slot dispatcher task, the current tree ships five adjacent simplifications. None regress any benchmark; two improve them.

  • Unified FxHashMap — dropped the custom IdentityU64Hasher in slot.rs in favor of the rustc_hash::FxHashMap already used by trie.rs. One hasher strategy across the crate, ~30 lines gone, fragile "one-u64-only" invariant gone. queue_group_4_members: −9.5 %, wildcard_routing: −9.0 % vs the prior snapshot (not shown in the big table because the big table compares to the dispatcher-era baseline).
  • Env-driven server config — HERMES_ADDR and HERMES_ROUTER_CAPACITY replace hardcoded [::1]:50051 and 8192. Aligns with the existing HERMES_* env convention.
  • Graceful shutdown — Server::serve_with_shutdown + tokio::signal::ctrl_c. Completes in-flight RPCs instead of dropping connections on Ctrl+C.
  • Unified delivery forwarding — grpc.rs now drives fanout and queue-group subscribers through one tokio::spawn loop via a DeliveryRx enum, instead of two near-identical spawn blocks. ~30 lines of duplication removed, Lagged handling centralized.
  • Session-scoped cleanup — new RouterCmd::SessionEnd { session_id }. The server allocates one SessionId per gRPC subscribe stream and sends a single cleanup command on close instead of tracking a Vec<SubId> and firing N Disconnects. SlotMap maintains a session_id → [sub_id] index.

Small-N trade-off

For 1–10 fanout subs the absolute latency grew a few hundred ns: the previous dispatcher pattern let the router and the dispatcher task run on two different tokio worker threads, so while the router prepared publish N+1, the dispatcher was waking subscribers for publish N. The direct-send design does both inline on the router, so small-N loses that parallel overlap. For large N the per-wake cost dominates and removing the hop wins decisively (13.9× at 1000 subs).

Methodology

  • Criterion 0.8, 100 samples per measurement, 3 s warmup + 5 s collection.
  • Each iteration: publish one message, synchronize on a sequence tracker via AtomicU64 + Notify, then yield.
  • All tasks spawned on the same multi-threaded tokio runtime (Criterion's default async_tokio).
  • Payload: 128 bytes. Latency is pure in-process router cost — no gRPC, no TLS, no TCP.
  • Both fanout_e2e (waits for all N subscribers to read) and fanout_publish_only (waits for a single sentinel subscriber, ignores N background drains) were run so publisher-side cost can be separated from consumer-side wake-up cost.
  • Test suite: 19 unit tests (adds session_end_removes_all_subs on top of the dispatcher-era suite), 6 integration tests, 1 doctest — all pass.
  • cargo clippy --workspace --all-targets -- -D warnings: clean.
